# V1.2D-C Wake Command Bridge 验证

状态：WakeEvent -> Command Recorder 模拟桥接已落地；未接入正式 `voice_overlay.py` 唤醒链路
日期：2026-05-03
范围：接入前状态机和 fake command recorder 验证

## 1. 目标

V1.2D-C 的目标是在不修改正式主链路的前提下，验证“收到 `WakeEvent` 后如何安全进入命令录音入口”的状态机。它承接 V1.2D-B 的 adapter 麦克风生命周期验证，但不打开真实麦克风、不启动 openWakeWord、不启动 STT server、不启动 `voice_overlay.py`。

## 2. 为什么需要桥接层

openWakeWord adapter 只能证明 wake listener 可以产生 coalesced `WakeEvent`，但正式接入还需要回答：

- 一次 wake event 是否只启动一次 command recorder。
- command recorder 运行期间，后续 wake event 是否会被忽略。
- TTS 播放期间，wake event 是否会被屏蔽，避免自唤醒。
- recorder 抛错时，bridge 是否能释放 busy 状态并进入安全状态。
- command 完成后，wake engine 是否可以恢复。

这些规则不应该直接散落在 `voice_overlay.py` 中。先沉淀为独立 bridge service，才能降低后续接入风险。

## 3. 当前仍不改 voice_overlay 的原因

`scripts/voice_overlay.py` 同时承载 GUI、STT server health、STT 文本唤醒、VAD 命令录音、LLM/TTS、conversation session 和 shutdown guard。直接改主链路会把麦克风占用、重复触发、TTS 自唤醒、会话状态和 fallback 风险混在一起。

V1.2D-C 只做 fake 状态机和模拟脚本，让正式接入点在 V1.2D-D / V1.2E 之前保持可验证。

## 4. 桥接状态机

新增文件：

```text
src/xiaohuang/wake_command_bridge_service.py
```

核心类型：

- `WakeBridgeDecision`
- `WakeCommandBridgeConfig`
- `WakeCommandBridgeState`
- `WakeCommandBridge`
- `FakeCommandStarter`

决策 reason：

| reason | 含义 |
|--------|------|
| `accepted` | wake event 被接受，fake command starter 被调用 |
| `disabled` | bridge 被配置禁用 |
| `cooldown` | 距离上一次 accepted event 未超过 `post_wake_cooldown_seconds` |
| `command_active` | 命令录音期间拒绝新 wake event |
| `tts_active` | TTS 播放期间拒绝新 wake event |
| `bridge_busy` | bridge 正在启动 command recorder 时拒绝新 event |
| `invalid_event` | 输入不是有效 `WakeEvent` |
| `recorder_error` | fake command starter 抛错，busy 状态已释放 |

`WakeCommandBridge` 不自动启动真实 recorder；它只调用传入的 `command_starter(event)`。正式接入时，上层仍需要明确何时 `mark_command_started()`、`mark_command_finished()`、`mark_tts_started()` 和 `mark_tts_finished()`。

## 5. FakeCommandRecorder 验证方式

`FakeCommandStarter` 记录调用次数和收到的 `WakeEvent`，也可通过 `raise_on_start=True` 模拟 recorder 启动失败。

单元测试覆盖：

- 首个 wake event accepted。
- cooldown 内重复 event 被拒绝。
- cooldown 后 event 可再次 accepted。
- command active / TTS active / disabled 状态拒绝 event。
- recorder error 后 `bridge_busy=False`。
- `reset()` 清空状态。
- fake starter 只收到 accepted event。

## 7. 接入 voice_overlay 前必须满足

- wake event 只启动一次 command recorder。
- command recorder 期间 wake engine pause 或 ignore。
- TTS 期间 wake engine pause 或 ignore。
- command 完成后恢复 wake 监听。
- recorder / adapter 出错时 fallback 到 `stt_text` 或给出清晰错误。
- 控制面板 READY 不能和 Wake Engine READY 混淆。

## 8. 风险

| 风险 | 说明 |
|------|------|
| 麦克风占用 | fake bridge 不证明真实 wake listener 与 VAD recorder 的设备切换 |
| 重复触发 | cooldown 和 command_active 必须同时存在，否则一次喊话可能启动多次录音 |
| TTS 自唤醒 | TTS 播放和播放后窗口需要 pause/cooldown |
| 会话状态冲突 | conversation session 可能已经在 follow-up 录音状态 |
| fake 与真实流程差异 | fake command starter 不覆盖真实录音失败、STT timeout 或 UI shutdown |

## 9. 下一步建议

- V1.2D-D：只读分析 `voice_overlay.py` 中 command recording 入口，设计最小接入点和 feature flag，不改主链路。
- 或 V1.2E：behind feature flag 接入 openWakeWord，并保留 `stt_text` fallback。

在完成 V1.2D-D 接入点设计前，不建议直接修改 `voice_overlay.py`。
