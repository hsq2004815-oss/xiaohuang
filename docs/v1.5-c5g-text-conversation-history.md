# V1.5 C5G Text Conversation History

## Scope

C5G adds local text conversation history to the web control panel. It is UI
history, not long-term LLM memory: persisted messages are rendered back into
the control panel, while the live LLM call still uses the existing
`TextInteractionSessionStore` / `session_id` path.

The implementation also adds a safe LLM configuration summary for diagnosing
`rule_fallback_no_key` without printing API keys.

## Storage

Runtime data is stored under:

```text
data/conversations/conversations.sqlite3
data/conversations/conversations.sqlite3-wal
data/conversations/conversations.sqlite3-shm
```

These files are ignored by Git.

Tables:

- `conversations` — conversation metadata, title, preview, message count.
- `messages` — user / assistant / system messages plus JSON snapshots.
- `conversation_multica_tasks` — conversation to Multica issue/run bindings.

## Request Contract

`send_text_message` accepts:

- `text` — user message text.
- `session_id` — legacy LLM session id.
- `conversation_id` — selected persistent conversation id.

Normal control-panel use sets:

```text
session_id = selected conversation id
conversation_id = selected conversation id
```

Compatibility:

- If `session_id` is empty, the LLM turn falls back to `control_panel`.
- The backend LLM call still uses `session_id`.
- Persistence uses `conversation_id`.
- `conversation_id` does not replace `text`, `session_id`, config, or any
  legacy message fields.

## Control Panel Actions

- `+ 新对话` creates a new persistent conversation.
- Selecting a conversation loads messages from SQLite.
- `清空会话` clears only messages in the selected conversation and preserves
  Multica bindings.
- `清除全部` asks for browser confirmation, then deletes all conversations,
  messages, and Multica bindings, clears in-memory text sessions, and creates
  a new blank conversation.
- `配置摘要` shows a safe LLM summary:
  `config_path`, `llm_configured`, `provider`, `model`, `api_key_present`,
  `env_key_name`, `env_key_present`, `key_source`.

## `rule_fallback_no_key` Diagnosis

`rule_fallback_no_key` is returned by `generate_llm_reply_result` when
`LlmReplyConfig.is_configured` is false. `run_text_interaction_turn` reports
`llm_configured` as:

```text
app_config.llm.enabled and llm_config.is_configured
```

If text chat shows:

```text
source: rule_fallback_no_key
llm configured: no
```

check the "配置摘要" output:

- `config_path` should point to the intended config file.
- `provider` / `model` should match `config.json`.
- `env_key_name` should match the configured provider.
- `env_key_present` should be `true`.

`scripts/control_panel_web.py` loads `%USERPROFILE%\.xiaohuang\secrets.ps1`
into the webview process on startup. It imports environment variables but never
prints key values.

## Safety Boundaries

- No real API key is stored in `config.json`, SQLite, logs, or docs.
- Debug summary exposes only boolean key presence and the environment variable
  name.
- "Clear all" deletes local control-panel history only; it does not touch
  `E:\DataBase`, logs, recordings, model caches, or external services.
- No shell execution is used for conversation history operations.
