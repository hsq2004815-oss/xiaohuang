## 1. 本次目标

V1.2B 的目标是新增一个可独立运行的 openWakeWord demo 入口，用来验证后续 Wake Engine 接入前的本机可行性：

- V1.2B-1 在 demo 层增加 `--cooldown-seconds` / `--no-coalesce`，用于把连续帧级 raw detection 合并成少量 coalesced wake event。

本阶段不修改 `voice_overlay.py`，不修改 wake/session/TTS/LLM 主链路，不启动 STT server，不训练中文“贾维斯”模型。

```text

```

`hey_jarvis` 可触发：

```text

```

命中 score 多次达到 `0.920` 到 `0.999`，最高接近 `0.998`。

```text
frames=748

```

重复唤醒测试：

```text
frames=373
detections=29
```

这说明英文 `hey_jarvis` 模型在本机可稳定触发。

## 8. detection 与 wake_event 合并解释

`detections=29` 不代表用户喊了 29 次。openWakeWord 是连续音频帧级检测；用户一次喊话时，多个相邻 frame 都可能 `score >= threshold`，因此 raw detections 会多于实际喊话次数。

正式接入小黄前，不能让每个 raw detection 都进入命令录音。V1.2B-1 已在独立 demo 中加入 per-label cooldown 合并：

- 默认 `--cooldown-seconds 2.5`。
- 默认开启 coalescing。
- `--no-coalesce` 可恢复每个 raw detection 都输出 wake_event 的实验行为。
- 同一 label 在 cooldown 窗口内只输出一次正式 `wake_event`，后续 frame 计入 `suppressed_detections`。

运行结束时 demo 输出：

```text
frames=<n>
raw_detections=<n>
coalesced_events=<n>
suppressed_detections=<n>
cooldown_seconds=<value>
```

预期真人测试中，`coalesced_events` 应明显少于 `raw_detections`，`suppressed_detections` 应大于 0。

## 10. CPU / 资源观察方法

后续真人体验时建议用以下方式观察资源：

- 先运行 `--duration-seconds 30 --debug --cooldown-seconds 2.5` 观察 score、raw_detections 和 coalesced_events。
- 同时用 Windows 任务管理器查看 `python.exe` CPU 和内存。
- 记录是否出现麦克风独占、设备 busy、音频 underrun/overflow。
- 不把运行日志、音频、模型文件提交进仓库。

建议命令：

```powershell
& "F:\for_xiaohuang\conda310\python.exe" scripts\wake_engine_demo.py --engine openwakeword --duration-seconds 30 --device 0 --debug --cooldown-seconds 2.5
```

如果需要更接近 16kHz 设备，可尝试 `--device 20`，但实际设备编号需以 `--list-devices` 输出为准。

## 12. 结论与下一步

结论：

- V1.2B 本机可行性验证通过。
- `device 0 + sounddevice` 可用。
- openWakeWord `hey_jarvis` 英文模型可稳定触发。
- 静默测试在当前 threshold 下误触发为 0。
- raw detections 多于实际喊话次数，必须用 cooldown / debounce / one-shot 合并。
- 当前不是中文“贾维斯”模型。

V1.2C 建议：

1. 实现 `WakeEngine` abstraction。
2. 将 event coalescing 纳入正式 service / adapter 边界。
3. 保留 `stt_text` fallback。
4. 在接入 `voice_overlay.py` 前，验证麦克风释放、命令录音切换、TTS 后 cooldown 和 rollback。

不建议在完成 V1.2C 抽象和安全验证前修改 `voice_overlay.py`。
