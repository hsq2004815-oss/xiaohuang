# V1.2C Wake Engine Service 设计

状态：V1.2C 服务抽象层已落地
日期：2026-05-03
范围：新增可测试的 WakeEngine service，不接入正式 `voice_overlay.py` 唤醒链路

## 1. 目标

V1.2C 将 V1.2B openWakeWord 独立 demo 中验证过的事件模型和 cooldown 合并逻辑下沉到 `src/xiaohuang/wake_engine_service.py`。本阶段只提供稳定、可测试、可扩展的服务接口，为 V1.2D 接入前的安全验证做准备。

本阶段不训练中文“贾维斯”模型，不下载模型，不启动 STT server，不调用 LLM/TTS，不修改 PowerShell，不修改 `voice_overlay.py`、`wake_loop_service.py` 或 `wake_word_service.py`。

## 2. 为什么需要服务抽象

V1.2B 的 demo 已证明 openWakeWord 在本机可以运行，但 demo 代码不能直接成为生产唤醒链路。正式接入前需要先把几个概念固定下来：

- wake event 的字段和含义。
- 引擎状态的可观测字段。
- raw detection 到 wake event 的 per-label cooldown 合并规则。
- 不依赖麦克风和 openWakeWord 的 fake engine 测试面。
- 后续 openWakeWord、Porcupine、FunASR KWS 或 STT text fallback 的统一边界。

## 3. WakeEvent 字段

`WakeEvent` 表示一次可被上层消费的唤醒事件。

| 字段 | 说明 |
|------|------|
| `engine_type` | 引擎类型，例如 `openwakeword`、`stt_text`、`porcupine`、`fake` |
| `wake_phrase` | 用户看到的唤醒词，例如 `贾维斯` |
| `label` | 模型实际 label，例如 `hey_jarvis` |
| `score` | 模型或匹配分数；无分数时可为 `None` |
| `detected_at` | 检测时间戳 |
| `raw_event_count` | 当前合并窗口内的原始 detection 数 |
| `suppressed_event_count` | 当前合并窗口内被 cooldown 抑制的 detection 数 |

## 4. WakeEngineStatus 字段

`WakeEngineStatus` 表示引擎当前状态，供未来控制面板或 adapter 查询。

| 字段 | 说明 |
|------|------|
| `engine_type` | 引擎类型 |
| `running` | 是否已启动监听或模拟运行 |
| `ready` | 是否可产生 wake event |
| `model_loaded` | 模型或 fake engine 是否已准备好 |
| `wake_phrase` | 当前显示唤醒词 |
| `sensitivity` | 当前阈值或灵敏度 |
| `last_wake_time` | 最近一次被接受的 wake event 时间 |
| `last_score` | 最近一次检测分数 |
| `error` | 错误摘要；正常时为 `None` |

## 5. WakeEventCoalescer 规则

`WakeEventCoalescer` 的规则是 per-label cooldown：

- 每个 label 独立记录最近一次被接受的 wake event 时间。
- 同一 label 在 cooldown 窗口内，第一次 detection 被接受，后续 detection 被 suppress。
- 不同 label 不互相抑制。
- cooldown 结束后，同一 label 再次 detection 会被接受。
- `stats()` 返回 `raw_detections`、`coalesced_events`、`suppressed_detections` 和 `cooldown_seconds`。
- `reset()` 清空 label 状态和统计。

V1.2B 的 demo 已改为复用该 coalescer；`--no-coalesce` 仍保留，每个 raw detection 都会作为 wake event 输出。

## 6. FakeWakeEngine 用途

`FakeWakeEngine` 用于单测和未来接入前模拟：

- 不打开麦克风。
- 不依赖 openWakeWord、sounddevice 或 numpy。
- 支持 `start()` / `stop()` / `status()`。
- `emit_fake_event()` 会走同一套 coalescer，便于验证 cooldown 行为。
- `set_error()` 可模拟 adapter 错误。

## 7. openWakeWord Adapter 边界

V1.2C 暂未新增 `openwakeword_adapter.py`。后续如新增 adapter，必须满足：

- optional import，导入 adapter 本身不能因缺 openwakeword 崩溃。
- 不在 import 时打开麦克风、下载模型或启动后台线程。
- 不启动 STT server、`voice_overlay.py`、LLM 或 TTS。
- 只负责模型加载、预测归一化、状态报告和显式 runtime 调用。

## 8. 未来与 voice_overlay 的接入点

V1.2D 之前建议先完成 adapter 安全验证，再考虑把当前：

```text
STT wake window -> wake_word_service text match -> command VAD recording
```

替换为：

```text
WakeEngine event -> pause/release wake listener -> existing command VAD recording
```

接入时必须保留 `stt_text` fallback，并确保 wake listener 与命令录音不会同时占用麦克风。

## 9. 为什么本阶段不改 voice_overlay

`voice_overlay.py` 当前承载 GUI、STT server 健康检查、wake loop、命令录音、LLM/TTS、conversation session 和 shutdown guard。直接在本阶段接入 openWakeWord 会把麦克风占用、线程生命周期、TTS 自唤醒和回滚风险一次性叠加。

V1.2C 先把接口和测试面稳定下来，降低 V1.2D 修改主链路时的风险。

## 10. V1.2D 前置条件

- 明确 openWakeWord adapter 的 import/runtime 边界。
- 验证麦克风释放和命令录音切换。
- 验证 TTS 播放后的 cooldown / mute window。
- 验证 READY 状态下不会重复触发命令录音。
- 明确中文“贾维斯”模型来源或继续只作为英文 `hey_jarvis` smoke。
- 准备回滚到 STT text wake path 的开关。

## 11. 风险

| 风险 | 说明 |
|------|------|
| 麦克风占用 | continuous wake listener 可能与 VAD 命令录音抢设备 |
| TTS 自唤醒 | 播报声音可能被 wake listener 捕获 |
| 命令录音冲突 | wake event 后需要暂停或释放 wake listener |
| 中文模型未完成 | 当前 `hey_jarvis` 不是中文“贾维斯”模型 |
| 事件重复触发 | raw detection 是帧级命中，必须继续使用 cooldown/coalescing |

## 12. 回滚

V1.2C 没有修改正式唤醒链路，因此天然可回滚。V1.2D 接入时也必须保留 STT text fallback，出现麦克风占用、误触发或模型不可用时可以回到当前稳定路径。
