# V1.2 Wake Engine 设计

状态：V1.2A 设计完成；V1.2B 独立 demo 已验证；V1.2C 服务抽象层已落地；V1.2D-A adapter harness 已落地
日期：2026-05-03
范围：设计与服务抽象，不接入正式 `voice_overlay.py` 唤醒链路

## 1. 背景与问题

小黄当前已经完成配置中控层、LLM Provider Router、Settings UI、托盘启动控制和状态控制面板。真人验证显示主链路可以启动、唤醒、识别、回复和播放 TTS，但唤醒体验仍然不稳定：用户经常需要喊多次“贾维斯”才能进入命令录音。

当前唤醒依赖短录音窗口后的 STT 文本匹配。这条路径天然有几个问题：

- 它要等一段音频录完并完成 STT 后才知道是否唤醒，延迟和窗口边界都会影响命中率。
- 唤醒词很短，STT 对短词、噪声、说话速度和起音截断更敏感。
- 它不是连续帧级检测，可能错过用户开口时的关键音素。
- STT 文本匹配难以直接给出 wake score、灵敏度和误触发控制。

V1.2 的目标是引入专门的 Wake Word / KWS 引擎，提高唤醒灵敏度，同时保留当前 STT 文本唤醒作为 fallback，避免破坏已稳定的 STT / LLM / TTS / control panel 主链路。

## 2. 当前链路说明

当前 wake path：

```text
recording window
  -> STT server / SenseVoiceSmall
  -> text match wake.phrases / wake.aliases
  -> command record with VAD
  -> STT transcription
  -> LLM or rule fallback
  -> optional edge-tts playback
  -> conversation follow-up window
```

当前实现边界：

- `scripts/voice_overlay.py` 负责悬浮窗入口、唤醒循环和命令轮次编排。
- `src/xiaohuang/wake_loop_service.py` / `wake_word_service.py` 负责现有 STT 文本唤醒逻辑。
- `src/xiaohuang/launch_control_service.py` 和 `status_control_service.py` 负责启动状态和控制面板 readiness。
- V1.1.4D-B 之后，控制面板状态刷新已经改为后台线程，不应再为 V1.2A 修改。

## 3. 目标链路设计

目标 wake path：

```text
continuous microphone frames
  -> wake engine model
  -> WakeEvent(phrase="贾维斯", score=...)
  -> pause or release wake listener
  -> existing command record with VAD
  -> STT transcription
  -> LLM or rule fallback
  -> optional edge-tts playback
  -> cooldown
  -> resume wake listener
```

原则：

- Wake Engine 只负责“是否唤醒”和 wake score，不负责命令转写、回复、TTS 或会话状态。
- 现有 STT 文本匹配必须保留为 `stt_text` fallback。
- V1.2B 先做独立 demo，不接入生产唤醒。
- V1.2C 已先抽象服务边界，再在 V1.2D 前做安全验证。
- 任何第三方 SDK 只能进入 adapter，不直接塞进 `voice_overlay.py`。

## 3.1 V1.2C 服务抽象落地

V1.2C 新增 `src/xiaohuang/wake_engine_service.py`，把 demo 层已经验证的 wake event、引擎状态和 per-label cooldown 合并逻辑沉淀为正式服务层：

- `WakeEvent`：统一描述上层可消费的 wake event。
- `WakeEngineStatus`：统一描述引擎运行、ready、model_loaded、last score 和错误状态。
- `WakeEventCoalescer` / `WakeEventStats`：统一统计 raw detections、coalesced events 和 suppressed detections。
- `FakeWakeEngine`：用于单测和后续接入前模拟，不依赖麦克风或 openWakeWord。

`scripts/wake_engine_demo.py` 已改为复用服务层 coalescer，但仍保持独立 demo；本阶段没有接入 `voice_overlay.py`，也没有训练中文“贾维斯”模型。

## 3.2 V1.2D-A Adapter Harness 落地

V1.2D-A 新增 `src/xiaohuang/openwakeword_adapter.py`，把 openWakeWord runtime 从 demo 脚本中隔离到 adapter 层：

- `OpenWakeWordAdapter` 负责 model lifecycle、sounddevice frame loop、prediction score 归一化、`WakeEvent` callback 和 `WakeEngineStatus`。
- `scripts/wake_engine_demo.py` 的真实监听路径已改为走 adapter；`--help` / `--dry-run` 仍不触碰模型或麦克风。
- 本阶段仍不修改 `voice_overlay.py`、不替换 STT 文本唤醒、不训练中文“贾维斯”模型。

## 5. 候选方案评估

### 5.1 openWakeWord

定位：V1.2 首选开源可控路线。

关键信息：

- Python 包名 `openwakeword`，官方安装为 `pip install openwakeword`。
- Windows 上使用 ONNX Runtime；TFLite / LiteRT 在 Windows 不是主路线。
- 推理输入是 16-bit 16kHz PCM，推荐 80ms 的倍数帧。
- 内置 VAD 可降低非语音噪声误触发。
- 预训练模型包含英文 `hey jarvis`，但中文“贾维斯”不能直接等同于英文模型。
- 自定义中文 wake model 需要后续训练或生成模型，V1.2A 不做。

评估：

| 维度 | 结论 |
|------|------|
| 安装难度 | 中。Python 3.10 匹配项目环境，核心依赖是 `onnxruntime`、`scipy`、`scikit-learn`、音频采集依赖。需要独立验证 PyAudioWPatch / sounddevice 在本机麦克风上的稳定性。 |
| Windows 兼容性 | 较好。官方说明 Windows 只安装 ONNX Runtime；本机小黄也是 Windows + Python。 |
| API key | 不需要。 |
| 自定义“贾维斯” | 支持路线清晰，但需要自定义模型训练或模型生成。英文 `hey jarvis` 只适合 sanity smoke，不适合直接作为中文“贾维斯”方案。 |
| 本地离线 | 推理可本地离线；训练阶段可能依赖外部 notebook、TTS 或模型下载。 |
| license 风险 | 代码 Apache-2.0；数据库记录提示预训练模型为 CC BY-NC-SA 4.0，后续使用模型文件必须单独复核。自训练模型也要记录训练数据和模型授权。 |
| CPU/资源风险 | 设计目标适合持续监听，但仍需在用户机器上观察单模型 CPU、内存和麦克风占用。 |
| 适合性 | 适合 V1.2 直接作为主路线，先独立 demo，再 adapter 化。 |

V1.2 采用方式：

- V1.2B 只做 `scripts/wake_engine_demo.py` 独立体验。
- 先用官方英文 `hey jarvis` 或本地可用模型验证音频帧、模型加载、score、threshold、CPU。
- 中文“贾维斯”模型训练作为后续独立任务，不在 V1.2A/V1.2B 一开始承诺效果。

### 5.2 wyoming-openwakeword

定位：server 架构参考，不作为 Windows 原生默认引擎。

关键信息：

- 它是 openWakeWord 的 Wyoming protocol server。
- 支持本地运行 TCP server，也支持 Docker。
- 支持指定 `--custom-model-dir` 加载自定义 `.tflite` 模型。
- 官方仓库 license 为 Apache-2.0。

评估：

| 维度 | 结论 |
|------|------|
| 安装难度 | 中到高。项目本身可本地安装，但更偏 Home Assistant / Wyoming 生态。 |
| Windows 兼容性 | 不如直接 Python adapter 明确。Windows 可考虑 WSL/Docker，但这会增加部署层复杂度。 |
| API key | 不需要。 |
| 自定义“贾维斯” | 取决于 openWakeWord 自定义模型。server 只解决常驻和协议，不解决中文模型来源。 |
| 本地离线 | 可本地运行。Docker/WSL 路线会增加系统依赖。 |
| license 风险 | Apache-2.0；模型 license 仍随 openWakeWord 模型而定。 |
| CPU/资源风险 | 与 openWakeWord 类似，另有独立 server 进程开销。 |
| 适合性 | 借鉴“独立 wake server 常驻”和事件协议。V1.2 不默认引入 Wyoming 协议。 |

V1.2 采用方式：

- 参考 server 模式：独立进程、可查询状态、可发送 wake event。
- 小黄后续如做 `scripts/wake_engine_server.py`，优先设计一个小而清晰的本地 IPC/JSON 事件接口，不直接把 Wyoming 作为硬依赖。

### 5.3 Picovoice Porcupine

定位：商业级体验标杆，可选方案，不作为默认路线。

关键信息：

- 支持 Windows x86_64 / arm64、macOS、Linux、移动端、浏览器和嵌入式。
- 自定义 wake word 通过 Picovoice Console 生成 `.ppn`，模型按平台区分。
- 初始化需要 Picovoice AccessKey，AccessKey 需要保密。
- 本地 Wake-Word 项目是 Electron + React + Picovoice Porcupine，适合参考桌面应用架构，不适合直接迁移到小黄 Python 主链路。

评估：

| 维度 | 结论 |
|------|------|
| 安装难度 | 低到中。SDK 成熟，Node/Python 都可用，但需要账号和 AccessKey。 |
| Windows 兼容性 | 强。官方明确支持 Windows。 |
| API key | 需要 AccessKey，且必须走 secrets/env，不能写配置文件或日志。 |
| 自定义“贾维斯” | 支持 Console 自定义生成 `.ppn`，但依赖云端 Console，文件平台相关。 |
| 本地离线 | 推理在本地；授权/使用统计/账号限制需按官方条款复核。 |
| license 风险 | SDK/服务和模型是 Picovoice 专有条款；免费使用和商业使用受限。不能作为默认开源可控路线。 |
| CPU/资源风险 | 通常较低，是体验标杆；仍需实测。 |
| 适合性 | 适合作为对照体验或用户明确接受 AccessKey 时的可选 adapter，不作为 V1.2 默认方案。 |

V1.2 采用方式：

- 不默认集成。
- 设计 `WakeEngine` abstraction 时保留 `engine_type="porcupine"` 的扩展位。
- 若后续用户要求体验 Porcupine，必须先处理 AccessKey、许可和 secret 边界。

### 5.4 sherpa-onnx KWS

定位：中长期语音底座研究，尤其适合边缘部署和多能力统一。

关键信息：

- sherpa-onnx 支持离线 ASR、TTS、VAD、speaker、keyword spotting 等能力。
- KWS 文档描述的是 open vocabulary keyword spotting，可以指定关键词而不重新训练模型。
- 官方仓库支持 Windows、Linux、macOS、Android、iOS 等，并支持多语言 API。
- 官方 KWS 预训练模型包含中文和英文路线。

评估：

| 维度 | 结论 |
|------|------|
| 安装难度 | 中到高。能力面很广，模型和参数较多，需要单独验证最小 KWS 命令和 Python API。 |
| Windows 兼容性 | 官方支持 Windows，但小黄当前已使用 FunASR，直接引入 sherpa-onnx 会新增一套语音栈。 |
| API key | 不需要。 |
| 自定义“贾维斯” | open vocabulary KWS 可通过关键词文件和阈值调参尝试，无需重新训练；实际中文“贾维斯”效果需本机实测。 |
| 本地离线 | 支持本地离线。 |
| license 风险 | 仓库 Apache-2.0；具体模型 license 和下载来源必须逐项复核。 |
| CPU/资源风险 | 取决于 KWS 模型和解码参数；beam search / tiny ASR 模式可能比专用 wake model 重。 |
| 适合性 | 不作为 V1.2 第一落地，适合 V1.3+ 或与边缘部署、全离线语音底座一起研究。 |

V1.2 采用方式：

- 在设计中保留 `engine_type="sherpa"`。
- 后续可做独立对比实验：同一麦克风、同一“贾维斯”样本，与 openWakeWord 比较命中率、误触发和 CPU。

### 5.5 Mycroft Precise

定位：老牌开源 wake listener，研究价值大于落地价值。

关键信息：

- 官方描述为 RNN wake word listener。
- 设计目标主要是 Linux，官方文档明确偏 Debian/Ubuntu/Raspbian。
- 可训练自定义模型，但依赖和生态较老。

评估：

| 维度 | 结论 |
|------|------|
| 安装难度 | 高。源码、训练和运行依赖较旧。 |
| Windows 兼容性 | 弱。官方主要面向 Linux。 |
| API key | 不需要。 |
| 自定义“贾维斯” | 理论可训练，工程成本高。 |
| 本地离线 | 可本地离线。 |
| license 风险 | 开源项目，但模型和社区数据需单独看来源。 |
| CPU/资源风险 | 需要实测；老项目不代表低风险。 |
| 适合性 | 只研究架构和历史经验，不作为 V1.2 落地候选。 |

## 6. 推荐路线

推荐 V1.2 主线：

1. V1.2 优先 openWakeWord。
2. 保留当前 STT 文本匹配作为 fallback 和 rollback。
3. Porcupine 作为体验标杆或可选方案，不作为默认。
4. wyoming-openwakeword 只借鉴独立 wake server 架构。
5. sherpa-onnx KWS / FunASR KWS 作为中长期对比研究。
6. Mycroft Precise 只研究，不进入近期实现。

理由：

- openWakeWord 与小黄当前 Python 3.10 + Windows + 本地离线目标最接近。
- openWakeWord 可以先做独立 demo，不需要立刻改主链路。
- adapter 化后可以把第三方依赖隔离在 `openwakeword_adapter.py`。
- 当前用户痛点是唤醒灵敏度，不是协议互联或边缘部署；因此先追求最短可验证路径。
- STT 文本唤醒已经可用，必须保留为 fallback，避免引入新引擎后主体验倒退。

## 7. 推荐架构

新增模块建议：

```text
src/xiaohuang/wake_engine_service.py
src/xiaohuang/openwakeword_adapter.py
scripts/wake_engine_demo.py

future:
scripts/wake_engine_server.py
```

`src/xiaohuang/wake_engine_service.py`：

- 定义通用 `WakeEngine` 协议或基类。
- 定义 `WakeEvent`。
- 定义 `WakeEngineStatus`。
- 提供 `start()`、`stop()`、`status()`、`set_callback()` 或 `detect(callback)` 语义。
- 不 import openWakeWord / Porcupine / sherpa。
- 单元测试应可用 fake engine 覆盖。

建议模型：

```python
@dataclass(frozen=True)
class WakeEvent:
    engine_type: str
    wake_phrase: str
    score: float | None
    detected_at: float
    raw_label: str | None = None


@dataclass(frozen=True)
class WakeEngineStatus:
    engine_type: str
    running: bool
    ready: bool
    model_loaded: bool
    wake_phrase: str
    sensitivity: float
    last_wake_time: float | None
    last_score: float | None
    error: str | None
```

`src/xiaohuang/openwakeword_adapter.py`：

- 只负责 openWakeWord 适配。
- 加载模型、读取麦克风帧、调用 `Model.predict()`。
- 将第三方 score 转换为 `WakeEvent`。
- 将异常转换为 `WakeEngineStatus.error`。
- 不调用 LLM / TTS / command recorder。

`scripts/wake_engine_demo.py`：

- 独立运行体验和诊断。
- 参数建议：`--engine openwakeword`、`--model-path`、`--wake-phrase`、`--sensitivity`、`--device`、`--duration-seconds`、`--debug`。
- 输出 wake score、last_wake_time、CPU observation 指引。
- 不启动 STT server，不启动 voice_overlay，不调用 PowerShell。

`scripts/wake_engine_server.py`（后续）：

- 独立常驻 wake server。
- 提供 status 查询和 wake event 订阅。
- 可以用 stdin/stdout JSON lines 或 localhost TCP/HTTP。
- 只在独立 demo 和 abstraction 可靠后再考虑。

## 8. 状态模型

`WakeEngineStatus` 必须支持控制面板和日志读取：

| 字段 | 类型 | 含义 |
|------|------|------|
| `engine_type` | str | `stt_text` / `openwakeword` / `porcupine` / `sherpa` / `disabled` |
| `running` | bool | 后台监听循环是否运行 |
| `ready` | bool | 是否可检测唤醒 |
| `model_loaded` | bool | 模型是否加载成功 |
| `wake_phrase` | str | 当前目标唤醒词 |
| `sensitivity` | float | 灵敏度或阈值，建议统一显示为 0.0 到 1.0 |
| `last_wake_time` | float 或 None | 最近一次唤醒时间戳 |
| `last_score` | float 或 None | 最近一次命中的 wake score |
| `error` | str 或 None | 最近错误，正常时为 None |

状态规则：

- `running=False` 时 `ready` 必须为 False。
- `model_loaded=False` 时 `ready` 必须为 False，除非 `engine_type="stt_text"` 且 fallback 正常。
- `error` 不应包含真实 API key、模型缓存路径中的敏感 token 或音频内容。
- READY 显示不代表 STT server ready；Wake Engine ready 与 ControlPanel `can_wake_now` 是不同状态，后续控制面板需要分别展示。

## 9. 控制面板集成规划

V1.2A 不改控制面板代码。

后续可在控制面板增加：

- Wake Engine: running / ready / error
- Wake model: 模型名或模型路径 basename
- Wake phrase: `贾维斯`
- Sensitivity: 当前阈值
- Last wake score: 最近一次命中分数
- Last wake time: 最近一次命中时间
- Fallback mode: enabled / disabled

集成原则：

- `status_control_service.py` 后续只读取 WakeEngineStatus，不直接 import 第三方引擎。
- 如果 wake engine 是独立 server，控制面板只做短超时 status 查询，不阻塞 UI。
- control panel READY 逻辑不能被 wake engine 状态混淆：STT/overlay/health READY 是“主链路 ready”，Wake Engine ready 是“唤醒检测 ready”。

## 10. 配置规划

V1.2A 不改配置代码。

未来 `config.json` 可新增或扩展：

```json
{
  "wake": {
    "engine": "stt_text",
    "phrases": ["贾维斯"],
    "sensitivity": 0.5,
    "model_path": null,
    "fallback_enabled": true,
    "device_index": 0
  }

```

字段规划：

| 字段 | 说明 |
|------|------|
| `wake.engine` | `stt_text` / `openwakeword` / `porcupine` / `sherpa` |
| `wake.phrases` | 继续作为用户可配置唤醒词；openWakeWord adapter 初期只取第一个 |

| `wake.model_path` | 自定义模型路径，不能指向 `E:\DataBase` raw 项目作为运行依赖 |
| `wake.fallback_enabled` | Wake Engine 失败时是否回退 STT 文本唤醒 |
| `wake.device_index` | 唤醒监听麦克风；默认沿用 `audio.device_id`，但可单独覆盖 |

Porcupine 特殊配置后续必须只走 env/secrets：

- `PICOVOICE_ACCESS_KEY` 或用户自定义 env 名。
- 不写入 config、README、日志或 commit message。

## 12. 风险与回滚

主要风险：

- 唤醒误触发：需要 VAD、threshold、cooldown、possibly patience 策略。
- 唤醒漏检：需要用户环境样本、阈值调优、自定义中文模型。
- CPU 占用：持续监听必须观察 CPU、内存和线程退出。
- 麦克风设备占用冲突：wake listener 与 command recorder 不能同时争用同一设备。
- STT 录音冲突：唤醒后要暂停或释放 wake listener，再进入现有 VAD command record。
- TTS 播放再次唤醒：TTS 播放和播放后 cooldown 期间应暂停 wake engine 或屏蔽事件。
- Windows 打包：PyAudioWPatch / sounddevice / onnxruntime / model files 都要单独验证。
- 模型 license：openWakeWord 代码、预训练模型、自训练模型、训练数据必须分别记录授权。
- Porcupine secret：AccessKey 只能走 secrets/env，且不能作为默认依赖。

回滚策略：

- 配置层保留 `wake.engine="stt_text"`。
- 启动 Wake Engine 失败时不影响 STT server、voice_overlay、LLM、TTS。
- V1.2D 接入前必须有单测覆盖 fallback 选择和 adapter 异常。
- 控制面板显示 wake engine error 但不阻断主链路 READY。
