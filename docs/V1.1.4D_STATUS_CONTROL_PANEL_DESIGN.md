# V1.1.4D 基础状态 UI / 控制面板设计

> 阶段：V1.1.4D 设计文档  
> 类型：docs-only  
> 目标：让用户清楚知道小黄当前是否已就绪，什么时候可以说“贾维斯”唤醒。  
> 非目标：不实现代码、不做高级 HUD、不做开机自启、不改 `voice_overlay.py` 主链路。

## 1. 背景与目标

V1.1.4C 已经让托盘具备启动、停止、重启小黄的能力，并且启动/重启会等待 STT server、`voice_overlay.py` 和 `/health` 同时 ready 后再提示就绪。

当前真实体验问题是：STT server 首次加载 SenseVoiceSmall 模型可能需要十几秒到一分钟。只靠托盘 messagebox，用户不知道系统处于“正在启动”、“正在加载模型”、“overlay 已启动但 STT 未 ready”，还是“可以说唤醒词”。如果用户过早说“贾维斯”，系统还没进入可唤醒状态，就会表现为没有反应。

V1.1.4D 的目标是设计一个基础状态 UI / 控制面板，解决“看不见状态”的问题。第一版只追求实用、清楚、稳定：

- 显示小黄总状态：未启动、启动中、加载模型中、就绪、停止中、重启中、部分运行、异常。
- 显示 STT server、voice overlay、health/model_loaded 的独立状态。
- 明确显示是否可以说“贾维斯”。
- 提供启动、停止、重启、刷新、打开设置、打开日志目录。
- 关闭控制面板不停止小黄。

## 3. 开源项目 / 数据库代码资产复用评估

### 3.1 可以直接使用的成熟库

| 能力 | 当前状态 | 结论 |
|------|----------|------|
| Tkinter | Python 标准库；`voice_overlay.py` 和 `settings_ui.py` 已使用 | V1.1.4D 推荐使用，零新增依赖。 |

| urllib | `launch_control_service.check_stt_health()` 已使用 | 继续用于 `/health`，不新增 `requests`。 |
| sqlite3 | 标准库 | 可留给 V1.1.5+ 状态历史记录；V1.1.4D 不需要。 |
| FastAPI/healthcheck 模式 | 数据库 code asset 提供参考 | 只借鉴 health shape 和错误摘要，当前不引入 FastAPI。 |

本阶段不建议新增依赖。PySide6、WebView、Tauri、psutil、requests 都不是 V1.1.4D 的必要条件。

### 3.2 可以借鉴的实现方式

- operation lock / busy flag：复用 V1.1.4C `OperationGuard` 思路，控制面板启动/停止/重启时禁用按钮或显示“操作进行中”。
- readiness polling：复用 `wait_until_ready()` 的语义；UI 以轮询状态展示，不只等 messagebox。
- subprocess argv list + `shell=False`：继续由 `launch_control_service` / tray 层封装，UI 不拼字符串命令。
- healthcheck timeout：health 失败不能崩溃；展示 `unavailable/timeout/error`。
- status panel state machine：将状态计算放 `status_control_service.py`，UI 只渲染。
- log summary / recent error display：UI 只显示摘要，详细错误指向 `logs/`。
- Settings UI save-reload / validation / unknown fields preserve：控制面板不重做配置编辑，只打开 Settings UI。

## 4. 技术方案比较

### 方案 A：Tkinter 基础控制面板

优点：

- 当前项目已有 Tkinter，`settings_ui.py` 和 `voice_overlay.py` 都在使用。
- 无新增依赖，适合快速实现最小可用状态面板。
- 打包和运行成本低。
- 可以直接复用现有 Settings UI 的窗口、tab、messagebox、后台线程经验。
- 适合“状态清楚优先”的阶段目标。

缺点：

- 视觉普通。
- 周期刷新和后台线程要小心，不能在非 UI 线程直接改 widget。
- 后续如果要做高级桌面应用外壳，可能需要迁移 UI 层。

### 方案 B：PySide6 / Qt 控制面板

优点：

- UI 更现代。
- 状态卡片、表格、主题和系统托盘生态更完整。
- 后续做复杂桌面壳更自然。

缺点：

- 新增依赖重，打包体积明显增加。
- 当前项目已有 Tkinter，短期切换会扩大风险。
- 容易把 V1.1.4D 从“状态可见”扩大成“桌面应用重构”。

推荐：V1.1.4D 先使用 Tkinter。业务逻辑放 `status_control_service.py`，UI 只是展示和调用。这样未来要换 PySide6 / WebView / Tauri 时，只替换 `scripts/control_panel.py`，保留状态服务和 launch control 服务。

## 7. 状态计算规则

ready 的定义：

- STT server 进程被检测到。
- `/health` 返回 `ok=True`。
- `/health.status` 为 `ready` 或 `model_loaded=True`。
- `voice_overlay.py` 进程被检测到。

partial 的定义：

- STT server 和 voice overlay 只有其中一个存在。
- 或 STT health ready，但 overlay 不存在。
- 或 overlay 存在，但 STT health 不可用。

`can_wake_now=True` 的唯一条件：

```text
overall_status == READY
stt_running == True
stt_ready == True
stt_model_loaded == True
overlay_running == True
```

具体规则：

| 条件 | overall_status | can_wake_now | 顶部提示 |
|------|----------------|--------------|----------|
| `stt_running=False` 且 `overlay_running=False` | `NOT_RUNNING` | False | 小黄未启动。 |
| `stt_running=True` 且 health not ready | `STARTING` 或 `LOADING_MODEL` | False | 正在加载 STT 模型，请稍候，暂时不要说唤醒词。 |
| `stt_running=True` 且 health ready 且 `overlay_running=True` | `READY` | True | 已就绪，可以说“贾维斯”唤醒。 |
| `stt_running=True` 且 health ready 且 `overlay_running=False` | `PARTIAL` | False | STT 已 ready，但 voice overlay 未运行。 |
| `stt_running=False` 且 `overlay_running=True` | `PARTIAL` | False | 检测到不完整状态，建议重启小黄。 |
| stop 操作进行中 | `STOPPING` | False | 正在停止小黄。 |
| restart 操作进行中 | `RESTARTING` | False | 正在重启小黄。 |
| health check 抛异常且无进程 | `NOT_RUNNING` 或 `ERROR` | False | 未连接到 STT server。 |
| health check 抛异常但 STT 进程存在 | `STARTING` 或 `ERROR` | False | STT server 尚未 ready 或 health 异常。 |

UI 顶部状态必须以 `can_wake_now` 为核心，而不是只看进程存在。

## 9. 托盘集成设计

V1.1.4D 后续实现时托盘菜单新增：

- 打开控制面板

- 启动小黄
- 停止小黄
- 重启小黄
- 打开设置
- 打开日志目录
- 关于/状态
- 退出托盘

集成规则：

- 点击“打开控制面板”启动 `scripts/control_panel.py --config <当前 config>`。
- 控制面板关闭不停止小黄。
- 退出托盘不默认停止小黄。
- 控制面板和托盘共享同一个 `launch_control_service`。
- 避免托盘和控制面板同时执行启动/停止/重启。第一版可在各自进程内禁用按钮；后续应考虑跨进程 operation lock。
- 如果控制面板已打开，第一版可以允许重复打开，记录为后续优化；更好的做法是单实例或激活现有窗口。

## 10. 常驻运行 V1.1.5 后续规划

用户关心“为什么不能一直运行”。V1.1.4D 只解决可见状态，不实现常驻策略。V1.1.5 可以单独规划：

- 小黄后台常驻。
- STT server 常驻，避免每次冷启动加载模型。
- `voice_overlay.py` 常驻或隐藏。
- 控制面板关闭不影响服务。
- 托盘随系统启动。
- 暂停监听 / 恢复监听。
- 完全退出小黄。
- 开机自启开关。
- 区分四种退出语义：
  - 关闭控制面板。
  - 退出托盘。
  - 停止小黄服务。
  - 完全退出所有小黄进程。

V1.1.5 还需要重新审视跨进程 operation lock、服务健康监控、崩溃恢复、状态历史记录和用户可理解的退出语义。

## 11. 日志与错误显示设计

控制面板显示最近错误摘要，但不显示敏感内容。

允许显示：

- `STT health check timeout`
- `voice_overlay not detected`
- `PowerShell command failed, see logs/tray_app.log`
- `health_unavailable:URLError`

禁止显示：

- API key。
- `secrets.ps1` 内容。
- Authorization header。
- token / password。
- 完整环境变量。
- 真实用户私密语音文本。

详细错误指向 `logs/`，UI 只做摘要。日志写入应继续复用现有脱敏思路，例如 `tray_app._sanitize_process_output()` 对 `sk-...` 和 `DEEPSEEK_API_KEY` 做遮蔽。
