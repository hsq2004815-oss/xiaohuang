## 1. 版本定位

V1.4-A 是小黄项目的一次关键架构升级：从纯语音聊天助手，演进为具备**安全白名单能力调用层**的桌面 Agent 雏形。

在 V1.3 阶段，小黄已具备完整的语音唤醒 → STT → LLM 回复链路，但所有"工具请求"（如"打开日志""查看状态"）只能返回 `tool_unavailable`，无法真正执行。V1.4-A 引入了 Capability Router，以确定性中文关键词匹配 5 个安全白名单能力，使小黄首次能**通过语音命令执行本地操作**。

核心设计原则（参考 OpenClaw 思想，不照搬代码）：
- 能力调用 typed、显式、可测试
- 权限默认保守：fail closed，deny 优先
- 不允许模型自由执行系统命令
- 每个能力调用记录 runtime event
- 不做多 channel、不做 Docker sandbox、不做浏览器自动化

## 3. 安全边界

### 明确禁止

| 操作类别 | 状态 | 拒绝关键词示例 |
|----------|------|---------------|
| 任意 shell / PowerShell / cmd | 拒绝 | `powershell`, `cmd.exe`, `shell`, `执行命令`, `系统命令` |
| 文件删除 | 拒绝 | `删除文件`, `删除`, `rm -`, `rmdir`, `del ` |
| 进程终止 | 拒绝 | `taskkill`, `kill process` |
| 系统控制 | 拒绝 | `shutdown`, `reboot`, `format `, `regedit`, `注册表` |
| 浏览器自动化 | 拒绝 | `打开浏览器`, `browser automation` |
| 微信 / QQ 操作 | 拒绝 | `微信`, `wechat`, `qq消息`, `发微信`, `发qq` |
| 代码生成/执行 | 拒绝 | `opencode`, `写代码`, `爬取` |

### 安全设计原则

- **优先拒绝**：高危关键词在能力匹配前先检查，一旦命中直接拒绝
- **不接受任意路径**：所有路径由 handler 内部硬编码，不来自用户或 LLM 输入
- **fail closed**：`route_capability()` 返回 `can_execute=False` 时，不会触发任何执行
- **异常不崩主链路**：handler 异常被 `execute_capability()` catch，返回结构化 `LocalCommandResult(ok=False)`
- **日志不含敏感信息**：runtime event 的 `sanitize_dict` 过滤 `api_key`/`secret`/`password`/`token` 等字段

## 4. 主链路接入

`reply_pipeline_service.py` 的 `generate_reply_pipeline_result()` 新增 `_try_capability_route()` 前置调用：

```text
收到 command_text
    │
    ├── route_capability(command_text)
    │       │
    │       ├── is_task_request=True, can_execute=True
    │       │       → execute_capability()
    │       │       → 返回结果 (source="capability")
    │       │       → 不调用 LLM
    │       │       → 可选 TTS
    │       │
    │       ├── is_task_request=True, can_execute=False
    │       │       → 返回拒绝消息 (source="tool_denied")
    │       │       → 不调用 LLM
    │       │       → 可选 TTS
    │       │
    │       └── is_task_request=False
    │               → 返回 None，继续走下方原有流程
    │
    └── 原有 LLM / rule fallback 流程 (不变)
```

**保持不变的链路**：
- `enable_llm` / `enable_tts` 行为不变
- DeepSeek key 未配置时 `rule_fallback_no_key` 不变
- 旧 `task_router_service.route_task()` 接口仍可用，已有测试全部保持通过

## 5. Runtime Events

所有能力调用记录 runtime event，source 为 `capability_router`：

| event_type | 触发时机 |
|------------|---------|
| `capability_invoked` | 能力开始执行前 |
| `capability_completed` | 能力执行成功 |
| `capability_failed` | 能力执行失败或 handler 异常 |

event recording failure 不会影响能力执行结果（`try/except` silently pass）。

## 8. 后续路线建议

### V1.4-B — Conversation Session Memory

当前每轮对话独立处理，无上下文。建议：
- 会话内保持简单的 turn history（用户说了什么、系统做了什么）
- 短时记忆窗口（如最近 3 轮）
- 不影响 LLM 单句回复模式

### V1.4-C — Capability Result UX

当前能力执行结果仅通过 TTS 播报和 overlay 显示。建议：
- 控制面板操作历史/"最近事件"展示能力执行结果
- 诊断导出 TXT 包含 capability 执行历史
- Web 控制面板新增"能力调用日志"视图

### 暂缓

| 项目 | 原因 |
|------|------|
| 任意命令执行 | 安全风险，当前不开放 |
| 浏览器自动化 | 复杂度高，安全边界不清 |
| 微信 / QQ 操作 | 平台限制、账号风险 |
| 插件系统 | 当前 5 个白名单足够验证架构 |
| LLM function calling | 当前确定性关键词匹配可满足 MVP 需求 |
| Docker sandbox | 引入重依赖，不符合桌面助手定位 |
| WebSocket Gateway | 当前本地单机架构不需要 |
