## 1. 当前结论

**当前 `--conversation-session` 是一个连续监听 + 自动退出的状态机，不是短期记忆系统。**

具体来说：
- **不保存** user/assistant turn 历史
- **不传** 历史上下文给 DeepSeek（`build_openai_compatible_chat_request` 只接收单个 `user_text`）
- **没有** `session_id` 或任何会话标识符
- `max_turns` / `session_timeout` / `followup_timeout` / `max_session_seconds` / `max_no_speech_retries` **真实生效**，控制何时退出连续监听
- **没有** runtime events 记录 `session_start` / `session_turn` / `session_end`
- 每一轮都独立调用 `generate_reply_pipeline_result(command_text)`，command_text 只是当前这一句

总结：用户唤醒小黄后可以连续说多句话，小黄逐句回复，但**小黄不记得上一句说了什么**。

## 2. 当前链路图

```text
Wake (openWakeWord / stt_text)
  │
  ▼
VAD record command
  │
  ▼
STT transcribe → command_text
  │
  ▼
run_assistant_turn_from_command(command_text)
  │
  ├── generate_reply (via _generate_reply_pipeline_guarded)
  │     │
  │     ├── _try_capability_route(command_text)
  │     │     ├── 白名单命中 → execute_capability → source=capability
  │     │     └── 危险请求 → 拒绝 → source=tool_denied
  │     │
  │     └── (非 task) → LLM / rule fallback
  │
  ├── session_config.enabled?
  │     │
  │     ├── YES → run_session_followup_loop()
  │     │         │
  │     │         ├── record_followup(followup_timeout) → next_text
  │     │         ├── is_session_exit_text? → exit
  │     │         ├── no_speech? → retry / exit
  │     │         ├── max_turns? → exit
  │     │         ├── max_session_seconds? → exit
  │     │         └── otherwise → generate_reply(next_text) → loop
  │     │
  │     └── NO → handle_single_turn_reply_result → idle
  │
  └── TTS (optional) → back to wake waiting
```

## 5. 缺口

### 5.1 没有短期 turn store

每轮 `generate_reply(command_text)` 只接收当前文本，没有把上一轮的 `(user_text, reply_text, source)` 存下来供后续轮次使用。

### 5.2 没有把上下文传给 DeepSeek

`build_openai_compatible_chat_request(user_text, ...)` 只构建单条 user message，不拼接历史 messages。LLM 看到的始终是孤立的一句用户输入。

### 5.3 capability 结果不进入上下文

用户说"打开日志目录" → capability 执行成功 → 回复"日志目录已打开"。下一轮用户说"再打开一次" → 小黄不知道之前做了什么 → 走 LLM/rule fallback（可能乱回）。

### 5.4 没有 session runtime events

`run_session_followup_loop()` 只在 logger 写日志，没有通过 `capabilities/runtime_events/service.py` 记录结构化事件。控制面板和诊断导出 TXT 看不到 session 级别的统计。

### 5.5 危险跨轮指代无法防御

用户第一句："打开日志目录"（capability 执行）。第二句："把它删掉"。
当前系统会把"把它删掉"当作独立命令，route_capability 中"删"命中 `_DENIED_KEYWORDS` → 拒绝。但如果用户说"再打开一次"，系统不知道"它"指什么。

### 5.6 没有 context window 控制

如果加了历史上下文字段，需要控制：
- 最近 N turns（建议 6-10）
- 最大 token/字符数（建议 1500-2000）
- 超时或 session end 后清空

## 6. V1.4-B 建议实现边界

| 项目 | 说明 |
|------|------|
| 只做短期内存 | 只存活在当前 session 生命期内 |
| 不落盘 | 不写文件、不写数据库 |
| 不写 E:\DataBase | 不动外部知识库 |
| 不做长期用户画像 | 不跨 session 持久化 |
| 不做向量数据库 | 无需 embedding / RAG |
| 不让 DeepSeek 决定工具 | capability router 始终优先 |
| 最近 6-10 turns | 滑动窗口裁剪 |
| 最大上下文字数 1500-2000 | `_shorten_reply` 已有截断逻辑可复用 |
| 超时或 session end 即清空 | `AssistantSessionOutcome` 返回后 reset |

## 7. 推荐实现方案

### 最小结构

```text
src/xiaohuang/conversation_session/
  __init__.py    — re-exports
  models.py      — ConversationTurn, SessionContext dataclass
  store.py       — append_turn(), get_recent_turns(), clear_context(), build_chat_messages()
  service.py     — (可选，如果 store 逻辑简单可以不放这里)
```

### 建议接入点

1. **`conversation_session/store.py`** — 纯内存 store：`list[ConversationTurn]` + `append()` + `get_recent(n)` + `clear()`

2. **`reply_pipeline_service.py`** — `generate_reply_pipeline_result()` 新增可选参数 `context: list[ConversationTurn] | None = None`，传给 LLM 路径时拼接到 chat messages

3. **`overlay_loop_runtime_service.py`** — 在 `run_overlay_runtime()` 的 while 循环外层创建 `SessionContext`，每轮调用 `reply_pipeline` 时传入 context

4. **`build_openai_compatible_chat_request()`** — 新增可选 `messages_history: list[dict] | None` 参数，拼接 system + history + current user message

### 不应该改的文件

- `capabilities/local_commands/*` — 能力路由不变
- `openwakeword_adapter.py` — 唤醒不变
- `scripts/voice_overlay.py` — 薄入口不变
- `scripts/stt_server.py` — STT 不变
- `control_panel_web_service.py` — 控制面板不变
